Why devs choose Inngest over Trigger.dev

Same durability. Less lock-in.

Any language

Trigger.dev's SDK is TypeScript-only — Python runs only as a shelled-out script via a build extension, not a native SDK. Inngest ships first-class SDKs for TypeScript, Python, and Go, and runs anything else over plain HTTP.

No task deploy step

Trigger.dev builds and deploys your task code to its own runtime. Inngest invokes the functions already running in your app. Your existing deploy pipeline is the only one you keep.

Event-driven by design

Fan-out, waitForEvent, and batching are core primitives, not add-ons. One event can trigger many functions, and a function can pause for months waiting on another — at zero cost.

The same workflow, both ways

One is a deploy target. One is a function call.

Trigger.dev — write a task, deploy it to their runtime

// trigger/welcome.ts
import { task } from "@trigger.dev/sdk";

export const welcome = task({
  id: "welcome",
  run: async (payload) => {
    await sendWelcomeEmail(payload.email);
    await startTrial(payload.userId);
  },
});

# build + deploy the task to Trigger.dev's runtime:
$ npx trigger.dev@latest deploy

Inngest — served from the app you already deploy

// inngest/welcome.ts
export const welcome = inngest.createFunction(
  { id: "welcome", retries: 4 },
  { event: "user.signup" },
  async ({ event, step }) => {
    await step.run("email", () =>
      sendWelcomeEmail(event.data.email));
    await step.run("trial", () =>
      startTrial(event.data.userId));
  }
);
// serve() on your existing /api route. No deploy target.

Both give you retries and durability. Only one adds a second build-and-deploy pipeline to own.

FAQ

The honest questions

Is Trigger.dev a bad choice?

No. It's a well-built, open-source task runtime with a great dashboard. The difference is design center. Trigger.dev optimizes for TypeScript teams who want a managed place to run tasks. Inngest optimizes for teams who want durable execution across languages, triggered by events, without deploying their logic to a second runtime.

Do I have to deploy my code to Inngest?

No. Inngest calls the functions running in your existing application over HTTP (or a persistent Connect socket for servers). There's no separate build or task-deploy step. Your current CI/CD is the only pipeline you maintain.

We use Python and Go, not just TypeScript.

That's the main reason teams pick Inngest here. Trigger.dev's SDK is TypeScript-only — it can shell out to Python scripts through a build extension, but there's no native Python or Go SDK. Inngest has first-class Python and Go SDKs, and anything else can trigger and run over HTTP, so one orchestration layer spans your whole backend.

Can I migrate from Trigger.dev?

Yes, and incrementally. A Trigger.dev task maps to an Inngest function; the body of your run() becomes step.run() calls. Most teams move one workflow at a time and run both in parallel during the transition.
